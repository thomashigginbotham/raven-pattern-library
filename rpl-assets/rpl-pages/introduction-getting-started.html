If you aren’t familiar with Node, Sass, or task runners like Gulp, then some of the files in the pattern library may seem intimidating, but as long as you know a little HTML and CSS, you shouldn’t have any difficulty putting RPL to good use.

Folder Structure

Most of your time will be spent editing the source code for your website and its components. These files are found in the src directory Let’s take a look at how they’re organized.

The src/html Folder

Your HTML pages are found in the src/html directory. Out of the box, you should see an index.html file and a couple others.

Your index.html file is your home page. If you open it in a code editor, you will notice some strange HTML comments that start with the word build. These comments are parsed by a Gulp plugin called processhtml. You don’t have to know the specifics of how it works, but be aware that this is how RPL adds your reusable components into your pages.

Whenever your see an HTML comment that starts with build:include, it will be replaced with HTML from another file.

The actual components used by your HTML pages are placed in the src/html/components directory. Depending on how many components are used by your website, you may want to create additional subdirectories to help organize your components.

Just as with your HTML pages, your components can also include components using the same build:include HTML comments.

The src/styles Folder

RPL uses Sass, a CSS pre-processor.

If you don’t know Sass, don’t panic. Your CSS knowledge is more than enough to use RPL, and most of the Sass that you see out of the box is easily understood.

The main.scss file is used to import all the other Sass files. So anytime you create a new Sass file, make sure to update main.scss.

If you look in the partials directory, you will find more Sass files. Some of these files are used for general styling (e.g. resets, layout, etc.), and some are specific to your components. When you create a new component, it’s a good idea to create a new Sass file for it too.

Files in the src/styles/partials directory begin with an underscore. This is a common Sass convention for partial files and allows you to import them without the underscore or the .scss extension.

For you Sass afficionados, you can create your own variables, functions, and mixins, and add them to the modules directory.

Asset Folders

In addition to the src/html and src/styles directories, you will see directories for fonts, images, and scripts. Add appropriate files to these directories and reference them from your components and HTML pages.

See the Working with JavaScript page for more information about adding JavaScript files to your project.

The rpl-assets Folder

This is where you will find options for configuring RPL and adding custom pages to its interface. See Customization and Custom Pages for more details.

The rpl-src Folder

The pattern library application files are found in the rpl-src directory. You can ignore these files unless you plan on making changes to RPL’s core functionality.

If you are an Angular/TypeScript developer and are interested in extending the functionality of the pattern library, feel free to make changes, but no documentation is provided of yet.

Workflow

During development, you can edit your HTML and other assets, and the pattern library will automatically refresh in your browser every time you save. Try it out now by editing this page (rpl-assets/rpl-pages/introduction-getting-started.html). As soon as you save your changes, you should see the browser reload to show them.

If, for some reason, things aren’t working as expected, double-check your file names and paths. A simple typo could be the problem. If that doesn’t work, try stopping and restarting the server (ctrl + c then npm run serve).

When you are done developing, it’s time to generate your production files. Do this by stopping the development server if it is running (ctrl + c). Then run npm run build. This will cause a dist folder to be created at the root of the pattern library. It will contain your optimized, production-ready files.
